Why agent handoffs usually fail

The problem is not just context length. It is continuity across sessions, tools, and agents.

Typical workflow
  • Agent A discovers architecture and bug history
  • Session ends and high-signal context evaporates
  • Agent B repeats grep, diffs, and reconstruction work
  • Decisions survive only in chat scrollback
With lossless-claude
  • Raw messages persist in SQLite
  • DAG summaries keep the active window lean
  • Durable decisions are promoted into shared memory
  • Next agent resumes from stored project context

Agent handoff flow

Different agents, one memory runtime.

01

Agent A works

Claude or Codex explores the codebase, uses tools, and accumulates project-specific context.

02

Memory is persisted

Messages are stored, compacted into DAG summaries, and high-signal outcomes are promoted into durable memory.

03

Agent B resumes

A later session restores relevant summaries, searches promoted memory, and continues without reconstructing the same history from scratch.

Integration modes

Claude remains the first-class interface. Codex shares the same backend through a separate adapter path.

Recommended

Claude Code

Hook-driven restore, prompt hints, transcript ingestion, compaction interception, and MCP access.

  • Automatic restore
  • Automatic writeback
  • Automatic compaction
  • Highest integration fidelity
Codex support

lossless-codex

Wrapper path for Codex that restores memory, injects hints, records the turn, and compacts stored state after the turn completes.

  • Automatic restore
  • Structured live ingestion
  • Automatic post-turn compaction
  • Same SQLite memory backend
Fallback only

Plain Codex

MCP tools plus an advisory AGENTS.md prompt. Useful, but not equivalent to the wrapper.

  • Manual retrieval
  • Manual storage
  • No automatic writeback
  • No hook-equivalent enforcement

Install paths

Claude-first on the surface, shared memory underneath.

Claude Code

claude plugin marketplace add ipedro/xgh-marketplace
claude plugin install lossless-claude
lossless-claude install

Use this when Claude Code is your primary agent interface and you want full hook-driven behavior.

Codex

npm install -g @ipedro/lossless-claude
npm install -g @openai/codex
lossless-codex "your prompt"

Use this when you want Codex on the same shared-memory backend with automatic restore and writeback.

MCP tools

Shared memory is queryable and writable from the agent layer.

lcm_search Search episodic and promoted knowledge across the project memory runtime.
lcm_grep Find exact terms, identifiers, file paths, and message fragments in stored history.
lcm_expand Recover deeper detail from compacted history when summaries are too compressed.
lcm_describe Inspect a stored summary or file by id.
lcm_store Persist durable project memory such as decisions, fixes, and conventions.
lcm_stats Inspect memory coverage, DAG depth, and compression state.
lcm_doctor Diagnose daemon status, hooks, MCP setup, and summarizer mode.

Lineage

The architecture is inherited. The current runtime model is not the original one.

Inherited from lossless-claw

The DAG-based compaction architecture, lossless message retention, and the LCM memory model come from lossless-claw by Martian Engineering, grounded in the LCM paper by Voltropy.

This fork adds

  • Local daemon runtime and MCP tool surface
  • Auto provider resolution across Claude and Codex
