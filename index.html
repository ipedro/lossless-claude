How it works

Compaction runs automatically when Claude Code's context is nearly full — no manual intervention needed.

1

Messages arrive

Every message stored verbatim in a local SQLite database. Nothing discarded.

2

Leaf compaction

When context fills up, lossless-claude intercepts the compaction hook and summarizes message chunks using a headless Claude process (Haiku).

3

DAG condensation

Leaf summaries condense into higher-level nodes, forming a directed acyclic graph of your full project history.

4

Knowledge promotion

High-signal nodes — decisions, bug fixes, architecture — promoted to a persistent FTS5 knowledge store, searchable across sessions.

5

Context restoration

At session start, recent summaries + promoted knowledge are injected as context. Full project history, minimal tokens.

7 MCP tools, available in every session

Claude calls these directly — no slash commands, no manual invocation required.

lcm_grep Search conversation history by keyword or regex
lcm_expand Drill into a summary to recover original messages
lcm_describe Describe the current DAG structure
lcm_search Search episodic and promoted knowledge
lcm_store Write to the promoted knowledge store
lcm_stats Token savings and compression ratios
lcm_doctor Diagnostics — daemon, hooks, MCP, summarizer

What we inherited

The DAG-based compaction architecture, lossless message retention, and the LCM memory model — all from lossless-claw by Martian Engineering. The underlying theory comes from the LCM paper by Voltropy.

What this fork adds

  • Cross-session knowledge store — FTS5 SQLite, zero external deps
  • Smart promotion engine — detects decisions, bugs, architecture
  • Background daemon with lazy auto-spawning
  • Native Claude Code plugin — hooks, MCP, one install command
  • Headless Claude subprocess for summarization — no API keys needed
